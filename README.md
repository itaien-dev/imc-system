# מערכת ניהול סדנאות — עמותת מ.ל.א

מערכת ניהול משתמשים וסדנאות לעמותת מ.ל.א ("זו בחירתי"), שלב 1 — לפי האיפיון המקורי מ-29/08/2019.

ראו את מסמך **"הוראות פיתוח – שלב 1"** המלא לפירוט הארכיטקטורה, סכמת בסיס הנתונים, ולוגיקת השדות המחושבים.

## מבנה הפרויקט

```
imc-system/
├── backend/     — Node.js + Express + PostgreSQL (Knex)
└── frontend/    — React (Vite)
```

## הרצה מקומית

### דרישות מוקדמות
- Node.js 18+
- PostgreSQL 14+ (מותקן ומורץ מקומית, או דרך Docker)

### Backend

```bash
cd backend
npm install
cp .env.example .env   # ועדכון DATABASE_URL ופרטי החיבור
npm run migrate         # מריץ את כל המיגרציות
npm run seed             # (אופציונלי) טוען נתוני בדיקה
npm run dev               # מריץ עם nodemon על http://localhost:4000
```

משתמש מנהל לאחר seed: `admin@imc.org.il` / `admin123`
משתמש חבר לדוגמה: `dana@mail.com` / `member123`

### Frontend

```bash
cd frontend
npm install
cp .env.example .env   # ועדכון VITE_API_BASE_URL אם נדרש
npm run dev               # מריץ על http://localhost:5173
```

## הערה לגבי ORM

מסמך ה"הוראות פיתוח" ממליץ על **Prisma** כ-ORM. בפועל, בסביבת הפיתוח הנוכחית (sandbox עם הגבלות רשת), לא ניתן היה להוריד את ה-engine הבינארי של Prisma מ-`binaries.prisma.sh`, ולכן הפרויקט נבנה בפועל עם **Knex.js** — ספריית JS טהורה, ללא תלות בקובץ בינארי חיצוני, עם תפקוד מקביל (מיגרציות מתועדות, query builder, תמיכה מלאה בטרנזקציות).

אם בסביבת הפיתוח/הפרודקשן בפועל של העמותה אין הגבלת רשת כזו, ניתן לעבור ל-Prisma ללא שינוי משמעותי בסכמה — מבנה הטבלאות, השמות, והאילוצים תוכננו כך שיתאימו ישירות ל-schema.prisma מקביל. זו החלטה ארכיטקטונית פתוחה לדיון.

## מצב נוכחי (שלב 1)

הושלם ונבדק (curl + בדיקות שכבת ה-API):
- התחברות (JWT) + שכחתי סיסמה (טוקן בזיכרון, ללא ספק מייל מחובר עדיין)
- אזור אישי (self-service) עם חסימת שדות מוגנים בצד שרת
- גריד משתמשים עם חיפוש, סינון, ושדות מחושבים (חבר עמותה, גיל, כמות סדנאות וכו')
- כרטיס משתמש מלא לעריכת מנהל
- גריד סדנאות + כרטיס סדנה עם 3 טאבים (סטודנטים/אסיסטנטים/צוות) וקריטריון קבלה מחושב
- סגירת סדנה (טרנזקציה: סימון נוכחות + יצירת משתמשים מהרשמות)
- עמוד בית ציבורי + טופס הרשמה לסדנה (עם rate limiting)
- קליטת CSV (משתמשים וסדנאות) עם תצוגה מקדימה ואישור

טרם מומש (להמשך):
- חיבור בפועל לספק מייל (SendGrid/SES) — כרגע הקישור לאיפוס סיסמה נכתב ל-console בלבד
- מסך "כרטיס משתמש" לא עוצב גרפית במקור (רק לוגיקת backend + מימוש frontend בסיסי)
- בדיקות אוטומטיות (unit tests) לשדות המחושבים — נדרש לפי פתק 3 באיפיון, ראו "הוראות פיתוח" סעיף 5
